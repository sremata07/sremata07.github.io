## Goal Driven vs. Issue Driven

Final projects were managed using "Issue Driven Project Management", or IDPM. The idea of this type of management is to split every task relating to the project into issues, and each issue would have a specified assigned person who would tackle the issue. Each issue would have it's own GitHub branch, and when the issue is solved the branch would be merged and the issue would be closed. In other words, this is a type of goal driven project management. Projects can be a massive undertaking, where lots of things need to be accomplished in order to get it done. As we all know, taking small consistent steps towards a goal is easier than taking one large leap towards a goal. So IDPM is a way of splitting up these goals into small issues that each person on the project can work on. This way, the work on the project is organized, no one is working on the same thing at the same time, and by calling the goals "issues", in a way it recontextualizes it as something that HAS to be done, whereas a goal is just something that you could get done, at least that's how I see it.

I personally love goal driven mindsets, and seeing as how IDPM is goal driven management specialized for group coding projects, I will be using this kind of project management in the future. I have two other friends who are also both pursuing computer science, though in different aspects than I am. I have been wanting to do a group project together for a while, and now that I learned and practiced IDPM, I will definitely be proposing that kind of project management to them. Though I will not be able to confirm future events in this essay, those who are reading this will have to trust that this project went well.
